# verify コマンド仕様

## 1. Why：コマンドの目的と価値

`sync` や `build` の前にマニフェストの問題を早期検出し、ITA APIを呼び出す前にローカルで完結する検証ステップを提供することが目的です。

* **フェイルファスト**: APIリクエストを送る前にエラーを検出することで、不完全なリソースがITA上に作成される状態を防ぎます。
* **開発者体験**: `manifest.yaml` 編集中にエラー箇所と修正ヒントを明示し、試行錯誤のコストを下げます。
* **CI連携の起点**: パイプラインの最初のゲートとして `verify` を配置することで、問題のある変更をマージ前に検出できます。

---

## 2. What：仕様の詳細

### A. 検証スコープ

| 検証カテゴリ | 内容 |
| :--- | :--- |
| **構文チェック** | `manifest.yaml` が有効なYAMLであること |
| **スキーマ検証** | Pydantic定義のJSONスキーマに対して型・必須フィールドを検証 |
| **参照整合性** | `conductor.movements` に列挙されたMovement名が `movements` セクションに存在すること |
| **命名規則** | Movement名・Role名が `[a-z0-9_]` のみで構成されること |
| **変数プレフィックス** | 変数名が `{role名}_{変数名}` の形式に準拠していること（`build` との整合性確保） |

### B. 出力フォーマット

検証結果は標準出力へ行ごとに報告します。

```
✔ YAML構文: OK
✔ スキーマ検証: OK
✗ 参照整合性: movements[1].name "web_setup" が movements セクションに未定義
✗ 命名規則: movements[2].name "WebSetup" に大文字が含まれています
```

* エラーが1件以上ある場合: exit code `1`
* 全件パス: exit code `0`

### C. オプション

| オプション | デフォルト | 説明 |
| :--- | :--- | :--- |
| `--manifest` | `manifest.yaml` | 検証対象ファイルのパス |
| `--strict` | `false` | 警告レベルの項目もエラーとして扱う |

> **TODO:** `--strict` で対象とする警告項目の一覧を確定する（例: 変数プレフィックス未適用を警告にするかエラーにするか）。

## 3. How：実行フロー

```bash
$ exastro-cdk verify
# → manifest.yaml のスキーマ検証・構文チェックを実行して結果を報告
```

```bash
$ exastro-cdk verify --manifest path/to/manifest.yaml --strict
```

### 内部処理フロー

```
manifest.yaml を読み込む
    ↓
YAMLパース（構文チェック）
    ↓
Pydanticモデルへのバリデーション（スキーマ検証）
    ↓
参照整合性チェック（Conductor ↔ Movements）
    ↓
命名規則・変数プレフィックスチェック
    ↓
結果サマリ出力（OK / エラー一覧）
```
