# SYNCHRONIZATION_PHASE

## 1. フェーズ目標

FOUNDATION_PHASEで確立した `init` コマンドとプロジェクト構造を土台に、**宣言的定義（`manifest.yaml`）とExastro ITA上の実際の状態を自動的に同期する**ことを目標とする。

核心となる `sync` コマンドを実装し、「コードを書けばExastroに反映される」という体験を実現する。

### 対象スコープ
| 機能 | 内容 |
| :--- | :--- |
| **`sync` コマンド（作成）** | `manifest.yaml` に定義されたリソースをExastroへ新規作成 |
| **`sync` コマンド（更新）** | 既存リソースとの差分を検出し、変更を適用 |
| **APIとの通信ロジック** | ITA APIを介したリソース操作の実装 |
| **差分検出** | ローカル定義と現在のExastro状態の比較ロジック |

---

## 2. `sync` コマンド設計

### 2.1 基本的な動作フロー

```
$ exastro-cdk sync
```

1. `manifest.yaml` を読み込み、目標状態（desired state）を構築する。
2. Exastro ITA APIを呼び出し、現在の状態（current state）を取得する。
3. 差分（desired - current）を算出する。
4. 差分に基づいてCreate/Updateの操作をITA APIへ発行する。
5. 実行結果をサマリとして出力する。

### 2.2 リソースモデルと依存関係

Exastro ITA上のリソースは以下のような構造を持つ。**Movementとロールパッケージはそれぞれ独立したエンティティ**であり、両者を結びつける紐付けテーブルが別途存在する（交差テーブル構造）。

```
┌─────────────┐       ┌──────────────────────┐
│  Movement   │       │  ロールパッケージ      │
│  (実行単位)  │       │  (Ansible Role ZIP)   │
└──────┬──────┘       └──────────┬───────────┘
       │                         │
       └──────────┬──────────────┘
                  │
       ┌──────────▼───────────────┐
       │  Movement-ロール紐付け    │
       │  (交差テーブル)           │
       └──────────────────────────┘
                  │
       ┌──────────▼───────────────┐
       │  Conductor               │
       │  (MovementのIDが必要)     │
       └──────────────────────────┘
                  │
       ┌──────────▼───────────────┐
       │  代入値自動登録設定        │
       └──────────────────────────┘
```

### 2.3 同期対象リソースと操作順序

上記の依存関係を考慮し、以下の順序で操作する。

```
Movement 作成/更新
    ↓
ロールパッケージのアップロード
    ↓
Movement-ロールの紐付け    ← MovementのIDとロールパッケージIDの両方が確定してから
    ↓
Conductor 作成/更新        ← MovementのIDが確定してから
    ↓
代入値自動登録設定 更新
```

> **設計上の注意:** Movementとロールパッケージはそれぞれ独立して作成・更新できるが、紐付けの更新は両エンティティのIDが確定した後でなければ実行できない。`sync` エンジンはこの依存順序を解決してからAPIを呼び出す必要がある。

### 2.4 冪等性の保証

`sync` は何度実行しても同じ結果になる（冪等）ことを保証する。

| 状態 | 挙動 |
| :--- | :--- |
| リソースが存在しない | 新規作成（Create） |
| リソースが存在し差分あり | 差分のみ更新（Update） |
| リソースが存在し差分なし | スキップ（No-op） |

## 3. API通信ロジック設計

### 3.1 認証

- リフレッシュトークンからアクセストークンを取得するロジックを実装する。
- 認証情報は環境変数または設定ファイル（`.exastro/config`）から読み込む。
- トークンの有効期限切れを検知し、自動リフレッシュを行う。

```bash
# 環境変数での設定例
export EXASTRO_BASE_URL=https://your-exastro-instance
export EXASTRO_REFRESH_TOKEN=your_refresh_token
export EXASTRO_ORGANIZATION_ID=your_org_id
export EXASTRO_WORKSPACE_ID=your_workspace_id
```

### 3.2 クライアント実装方針

- `requests` ライブラリをラップした `ITAClient` クラスに通信ロジックを集約する。
- レスポンスのエラーハンドリングを統一し、リトライポリシーを設ける。
- 将来的なGoへの移行を見据え、HTTPクライアントの詳細をインターフェース越しに隠蔽する。

### 3.3 差分検出ロジック

- `manifest.yaml` から生成した内部表現（Pydanticモデル）とAPIレスポンスを正規化して比較する。
- 変更があったフィールドのみをパッチとしてAPIへ送信する。

## 4. 実装アイテムリスト

### 4.1 `sync` コマンド本体
- `exastro-cdk sync` コマンドのエントリポイント実装。
- desired state の構築（`manifest.yaml` → Pydanticモデル変換）。
- current state の取得（ITA API呼び出し）。
- 差分算出ロジックの実装。
- Create/Update操作の発行。
- 実行サマリの出力（作成数・更新数・スキップ数）。

### 4.2 ITA APIクライアント
- アクセストークン取得・リフレッシュ処理。
- Movement CRUD APIのラッパー実装。
- ロールパッケージ アップロードAPIのラッパー実装。
- Movement-ロール紐付けAPIのラッパー実装（交差テーブル操作）。
- Conductor CRUD APIのラッパー実装。
- 代入値自動登録設定 APIのラッパー実装。
- エラーハンドリングとリトライ処理。

### 4.3 状態管理
- リソース依存関係の解決（Movement / ロールパッケージ → 紐付け → Conductor の順序制御）。
- MovementとロールパッケージそれぞれのIDを独立して追跡し、紐付け操作時に結合する。
- 既存リソースIDの追跡（状態ファイルまたはAPIからの取得で管理）。

---

## 5. 完了条件 (Exit Criteria)

1. `exastro-cdk sync` を実行すると、`manifest.yaml` の内容がExastro ITAに反映される。
2. 差分がない場合はスキップされ、不要なAPIコールが発生しない。
3. 2回連続で `sync` を実行しても、結果が変わらない（冪等性の確認）。
4. 実行後のサマリ（作成・更新・スキップの件数）が標準出力に表示される。
